# Отчет по улучшению API (Модель зрелости Ричардсона)

## 1. Оценка текущего уровня
Текущая реализация соответствует **Уровню 1** (Resources). 
* У нас есть разделение на ресурсы, но методы используются неправильно (все через POST).
* Названия эндпоинтов содержат глаголы (`/add-room`, `/booking`), что противоречит стандартам REST.

## 2. Предложенные изменения (Переход на Уровень 2)
Для достижения уровня **HTTP Verbs**, я внес следующие изменения в контракт:

### А) Использование существительных вместо глаголов
* Было: `POST /add-room` 
* Стало: `POST /rooms` — мы создаем новый ресурс в коллекции комнат.
* Было: `POST /booking`
* Стало: `PATCH /rooms/{id}` — мы частично обновляем состояние конкретной комнаты (меняем статус брони).

### Б) Правильные HTTP методы и статусы
* **POST** используется только для создания.
* **PATCH** используется для изменения статуса `is_booked` (идемпотентность и семантика частичного обновления).
* Для успешного создания теперь возвращается статус **201 Created** вместо 200.
* Для получения данных используется **GET**.

### В) Структура URL
* Идентификаторы ресурсов теперь передаются в пути (Path Parameters), а не в теле запроса: `/rooms/1` вместо `{"roomId": 1}`. Это делает API интуитивно понятным и ресурсным.
